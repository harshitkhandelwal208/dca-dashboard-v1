1. System Overview

The team-event spreadsheet system converts Hill Climb Racing 2 team-event screenshots into structured Discord reports. A user posts one or more screenshots in a configured monitored channel. The bot groups the images into a session, sends them through Gemini Flash, parses the event data, applies staff corrections if any exist, and generates files.

Input

Team-event screenshots, including standings, podium screens, score summaries, and ranking lists.

Processing

Gemini extracts ranks, names, scores, event points, team labels, team scores, and visible metadata.

Output

The bot posts an XLSX workbook, a spreadsheet-style image, a chart image, and period reports.

Current visual behavior: event spreadsheet images show own-team rows only, hide unnecessary Team and Type columns, and use colorful metric formatting. Weekly reports use a blue-kill grid with one event/enemy column per included session.

Setup Requirements

These settings decide which screenshots are captured, who can manage them, and where final files are posted.

EnabledThe spreadsheet module and each individual spreadsheet team must be enabled before screenshots are captured.
Monitored channelThe Discord channel where users post event screenshots. Only images in this channel are captured for that team.
Output channelThe channel where final files and automatic reports are posted. If not set, output falls back to the submission channel for some flows.
Access roleMembers with this role can use spreadsheet commands for the team. Server administrators always have access.
Own team aliasesNames Gemini may see for the configured team. These help classify rows as own-team rows.
Own player aliasesKnown player names that should be treated as own-team players even if the screenshot row is ambiguous.
Auto processWhen enabled, the bot automatically processes the session after the grouping window ends.
Grouping windowThe number of minutes during which images from the same user and channel are grouped into one pending session.

Screenshot Submission Workflow

The cleaner the input screenshots are, the fewer corrections staff need to make afterward.

Post in the monitored channel

Only screenshots posted in the configured monitored channel are captured. Screenshots posted elsewhere are ignored by the spreadsheet system.

Use one Discord user per submission

The grouping logic groups images by user and channel. If multiple users post parts of the same event, the bot may create separate sessions.

Post all event screenshots close together

Images sent within the grouping window are appended to the same pending session. Waiting too long can split the event into multiple sessions.

Include useful screens

Best results usually come from final standings, team score summaries, podium screens, and clear rank/player/score rows.

Wait for processing

If auto-process is enabled, the bot runs after the grouping timer ends. Otherwise, staff can run /spreadsheets generate.

Screenshot Quality Checklist

Good Screenshots

  • Readable rank rows
  • Visible player names
  • Visible scores or event points
  • Visible own/enemy team context
  • All pages needed for the full event

Avoid

  • Unrelated images in the same grouping window
  • Heavy crops that hide ranks or names
  • Duplicate screenshots unless needed
  • Multiple users splitting one event submission
  • Posting a new event before the previous session is done
Practical rule: if staff would need the image to understand the event manually, include it. If it is unrelated to the event result, leave it out.

Event Outputs

Each processed event can produce a workbook, a clean visual table, and a chart.

XLSX Workbook

Full data output with Summary, Results, and Attendance sheets. This is the best artifact for archiving and detailed review.

Spreadsheet Image

A cleaner visual table for Discord. It is uploaded to Discord and then removed from local bot storage.

Chart Image

Compares own/enemy score, own/enemy event points, and placement buckets. It can be rebuilt from stored parsed session data.

Storage behavior: source screenshots are temporary processing inputs. Generated spreadsheet and report images are delivery files, so the bot deletes local copies after upload and prunes leftovers by local image retention.

Session Summary

The summary command shows the event name, session ID, parsed player counts, own/enemy score totals, top 10 counts, KAB players, and best own-team rows. It is the fastest way to review whether the bot understood the event before staff correct anything.

/spreadsheets summary team:<team> session_id:<id>

Weekly Reports

Weekly reports are built around blue kills and event attendance for a single team-event name.

The weekly report includes processed sessions from the selected week that match the same team-event name. If a player is known from the roster or previous own-team rows but misses an event, that event is scored as zero.

How To Read It

Enemy columnsEach enemy team column represents one included event/session. The number below the enemy name is the possible blues for that event.
Player cellsEach value is how many opponents the player ranked above in that event.
%KillTotal blues killed divided by total possible blues across the included events.
TotalThe player's total blues killed across the report.
MaxThe maximum possible blues for that player across attended and roster-counted events.
#KABThe number of events where the player killed all possible blues.
Ranking behavior: weekly report ranks are based primarily on total blues killed. Ties share the same rank when their total blue-kill count matches.

Monthly Reports and Attendance

Monthly reports provide broader performance tracking across all processed sessions in a month.

Monthly Report Focus

  • Total score
  • Total event points
  • Total blues killed
  • Blue kill percentage
  • KAB count
  • Missed and attended event counts

Attendance Logic

  • The bot learns own-team players from configured aliases and historical processed sessions.
  • Known players missing from an event receive zero for that event.
  • The Attendance sheet records attended and missed players.

Automatic Report Timing

WeeklyPosted automatically when a new processed session has a different event name than the previous processed session. This indicates the prior event has ended.
MonthlyGenerated during maintenance after month end. Staff can force monthly generation when needed.
Duplicate guardAutomatic report posting tracks emitted period keys so the same report is not posted repeatedly unless staff forces regeneration.
Anchor dates: report commands accept an optional anchor_date. Any date inside the target week or month can be used.

Command Reference

These are the main slash commands used by staff and admins.

Status and Session Commands

/spreadsheets status team:<team>

Shows team setup, monitored/output channels, access role, grouping window, Gemini settings, retention, output format, and session counts.

/spreadsheets sessions team:<team>

Lists the 10 most recent spreadsheet sessions for that team.

/spreadsheets summary team:<team> session_id:<id>

Shows parsed summary text for a pending, failed, or processed session.

Generation and Files

/spreadsheets generate team:<team> session_id:<id>

Processes or refreshes a screenshot session. If no session ID is given, it uses the latest matching session.

/spreadsheets generate team:<team> rerun_gemini:true

Downloads the source images and runs Gemini extraction again.

/spreadsheets file team:<team> session_id:<id>

Returns the generated XLSX or FODS spreadsheet file.

/spreadsheets chart team:<team> session_id:<id>

Returns the generated summary chart image.

Report Commands

/spreadsheets weekly team:<team> anchor_date:YYYY-MM-DD

Builds the weekly report for the week containing the anchor date.

/spreadsheets monthly team:<team> anchor_date:YYYY-MM-DD

Builds the monthly report for the month containing the anchor date.

/spreadsheets regenerate-weekly team:<team>

Regenerates and posts the weekly report to the configured output channel.

/spreadsheets regenerate-monthly team:<team>

Regenerates and posts the monthly report to the configured output channel.

Corrections and Rebuilds

Corrections are stored on the session, reapplied to parsed data, and then outputs are rebuilt.

/spreadsheets correct-name team:<team> session_id:<id> row:<rank> value:<name>

Corrects a parsed player name.

/spreadsheets correct-team team:<team> session_id:<id> row:<rank> team_type:Own team

Marks a row as own team.

/spreadsheets correct-team team:<team> session_id:<id> row:<rank> team_type:Opponent value:<team label>

Marks a row as opponent and optionally sets the opponent label.

/spreadsheets correct-placement team:<team> session_id:<id> row:<rank> placement:<new rank>

Corrects the placement/rank for a row.

/spreadsheets correct-points team:<team> session_id:<id> row:<rank> field:Score value:<score>

Corrects either score or event points.

/spreadsheets correct-event-name team:<team> session_id:<id> value:<event name>

Corrects the event title used in files and reports.

/spreadsheets rebuild team:<team> session_id:<id>

Rebuilds output files from existing parsed Gemini data and stored corrections.

Correction targeting: correction commands use the visible placement/rank row where possible. If no matching rank is found, the code falls back to row order. Always check the summary after correcting.

Generic Correction Command

The generic command can correct player name, team name, placement, points, score, team type, or event name.

/spreadsheets correct team:<team> session_id:<id> row:<rank> field:<field> value:<replacement>

Diagnostics, Troubleshooting, and Best Practices

Use these checks when a session does not look right or a report is missing.

Temporary Diagnostic Commands

/spreadsheets test-gemini team:<team> session_id:<id>

Runs Gemini extraction for a pending, processed, or failed session and returns a preview.

/spreadsheets test-grouping team:<team>

Shows pending sessions and grouping-window state.

/spreadsheets preview team:<team> session_id:<id>

Shows parsed output without posting final files.

/spreadsheets force-weekly team:<team>

Forces weekly report posting.

/spreadsheets force-monthly team:<team>

Forces monthly report posting.

/spreadsheets rebuild-event team:<team> session_id:<id>

Temporary alias for rebuilding one event output.

Common Problems

ProblemWhat To CheckLikely Fix
No session appearsWrong channel, disabled team, or no image attachmentsCheck /spreadsheets status and monitored channel config.
Images split into multiple sessionsPosted after grouping window or from different usersUse test-grouping, then generate the correct session manually.
Bad player/team classificationAmbiguous screenshot row or missing aliasesUse correct-team and update aliases for future events.
Bad event nameGemini read title incorrectlyUse correct-event-name, then rebuild/report again.
No report postsMissing output channel, no sessions, duplicate guard, or unavailable channelSet output channel or use regenerate commands with the right anchor date.
Enemy name includes scoresOCR merged team score text into the labelThe cleaner strips own team and score fragments, but staff can still correct team labels if needed.

Best Practices

For Users

  • Post all screenshots together.
  • Use the correct monitored channel.
  • Include clear standings and score screens.
  • Do not mix unrelated images into the same submission.

For Staff

  • Review summaries before trusting reports.
  • Correct names and team assignments early.
  • Regenerate reports after important corrections.
  • Keep own-player aliases current.
